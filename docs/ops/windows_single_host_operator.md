# Windows Single-Host Operator Guide

This document is the authoritative operator handoff for the supported Windows-first deployment target. It covers fresh install, upgrade, backup, restore, rollback, and secret rotation for the supported runtime surface:

- the FastAPI facade at `service.api_server`
- the installed `earctl` CLI for administrative tasks that do not depend on a source checkout

Use this guide for deployed hosts. `README.md`, `RUNBOOK.md`, `service/windows/*.md`, and `docs/api/readme.md` should defer to this file for lifecycle operations.

## Scope and support boundary

- Supported target: one Windows host running one EarCrawler API service.
- Supported release-grade deployment artifacts for the API service: signed release checksums plus the published wheel and `hermetic-artifacts.zip` (pinned lockfile + wheelhouse installer payload) installed into a dedicated Python virtual environment.
- Supported service wrapper: NSSM (`nssm.exe`) pointing at that virtual environment.
- Rate limits, concurrency controls, the RAG cache, retriever caches, and retriever warmup state are process-local. This guide does not claim multi-instance correctness.
- Not the operator deployment path: a source checkout, `earctl api start`, `scripts/api-*.ps1`, the PyInstaller `earctl.exe`, or the Inno Setup installer. Those are repo-local or convenience surfaces, not the authoritative API hosting path.
- Provision the local read-only Fuseki dependency first using `docs/ops/windows_fuseki_operator.md`. This API guide assumes the supported Fuseki endpoint is `http://127.0.0.1:3030/ear/query` unless you intentionally override it.
- Machine-readable runtime/capability state is reported in `/health` under `runtime_contract` and published in `docs/api/capability_registry.json`.
- If a deployment must accept traffic beyond the trusted local host boundary, keep this guide as the app baseline and add the IIS + ARR front-door pattern in `docs/ops/external_auth_front_door.md`. Do not expose the EarCrawler process directly with only `EARCRAWLER_API_KEYS`.

For the deferred future-work note on scale-out design, see
`docs/ops/multi_instance_deferred.md`.
For the runtime-state ownership boundary behind that contract, see
`docs/single_host_runtime_state_boundary.md`.

## Baseline operator profile

Treat the following as the supported default posture for deployed hosts unless a
later section explicitly calls out an optional validation mode:

| Surface | Supported baseline | Optional operator-controlled mode | Quarantined or not part of deployed-host baseline |
| --- | --- | --- | --- |
| Deployment shape | One Windows host, one API service instance | None | Multi-instance or load-balanced deployment |
| API scope | `/health`, `/v1/entities/{entity_id}`, `/v1/lineage/{entity_id}`, `/v1/sparql`, `/v1/rag/query` | `/v1/rag/answer` only with explicit LLM/runtime enablement | `/v1/search` |
| Retrieval mode | `dense` | `EARCRAWLER_RETRIEVAL_MODE=hybrid` | KG expansion is not baseline retrieval behavior |
| LLM posture | Remote answering denied by default | Remote provider answering or local-adapter serving with explicit env and evidence | Search/KG promotion by operator choice alone |
| Auth posture | Loopback/single-host plus static shared secret or anonymous lower-quota mode | Remote provider secrets only when optional answering is enabled | Broader internet-facing auth claims |
| Graph dependency | Local read-only Fuseki `5.3.0` service provisioned per `docs/ops/windows_fuseki_operator.md` | Reviewed alternate local port with matching `EARCRAWLER_FUSEKI_URL` override | Text-index-backed Fuseki search or any broader Fuseki promotion |

The capability registry and `/health` runtime contract should match this table:

- `api.default_surface`: `supported`
- `api.rag_answer`: `optional`
- `retrieval.hybrid`: `optional`
- `runtime.local_adapter_serving`: `optional`
- `api.search`: `quarantined`
- `kg.expansion`: `quarantined`

## Standard host layout

| Path | Purpose |
| --- | --- |
| `C:\Program Files\EarCrawler\runtime` | Dedicated runtime root |
| `C:\Program Files\EarCrawler\runtime\.venv` | Python virtual environment that hosts the wheel |
| `C:\ProgramData\EarCrawler\workspace` | Working directory for reports and runtime artifacts |
| `C:\ProgramData\EarCrawler\config` | Deployment-owned configuration files |
| `C:\ProgramData\EarCrawler\logs` | Service stdout/stderr logs |
| `C:\ProgramData\EarCrawler\backups` | Operator-created backup snapshots |

## Required inputs

Before touching the host, collect:

- the signed release wheel, for example `earcrawler-0.2.5-py3-none-any.whl`
- the release checksum file generated by `scripts/checksums.ps1`
- the release checksum signature (for example `checksums.sha256.sig`)
- `hermetic-artifacts.zip` from the same signed release payload
- the release validation evidence files when available:
  - `dist\api_smoke.json`
  - `dist\installed_runtime_smoke.json`
  - `dist\optional_runtime_smoke.json`
  - `dist\security\security_scan_summary.json`
  - `dist\observability\api_probe.json`
  - `dist\release_validation_evidence.json`
- Python 3.11 on the host
- NSSM installed on the host
- the Fuseki SPARQL query endpoint URL, for example `http://127.0.0.1:3030/ear/query`, provisioned per `docs/ops/windows_fuseki_operator.md`
- a recorded Task 5.3 adapter artifact only if optional local-adapter serving
  will be enabled during an explicit local-adapter review window; it is not
  part of the normal production-beta deployment path
- any required secrets:
  - `EARCRAWLER_API_KEYS` for inbound API auth if you are not running anonymously
  - `TRADEGOV_API_KEY` for scheduled ingest jobs
  - `GROQ_API_KEY` or `NVIDIA_NIM_API_KEY` only if optional remote answering is enabled

## Lifecycle automation scripts

The supported single-host lifecycle now has dedicated PowerShell automation under
`scripts/ops/`:

- `scripts/ops/windows-fuseki-service.ps1`
  - actions: `install`, `uninstall`, `start`, `stop`, `restart`, `status`, `health`, `render-config`
  - provisions the supported read-only Fuseki service wrapper and its assembler config
- `scripts/ops/windows-fuseki-backup.ps1`
  - creates a timestamped Fuseki snapshot under the backup root
  - writes `snapshot_manifest.json` and `checksums.sha256` for integrity evidence
- `scripts/ops/windows-fuseki-restore-drill.ps1`
  - runs a non-destructive restore drill from a Fuseki snapshot
  - verifies checksums and writes `restore_drill_report.json`
- `scripts/ops/windows-single-host-service.ps1`
  - actions: `install`, `uninstall`, `start`, `stop`, `restart`, `status`, `health`
  - enforces loopback (`127.0.0.1`/`localhost`) by default so operators do not
    accidentally claim unsupported multi-host deployment behavior
- `scripts/ops/windows-single-host-backup.ps1`
  - creates a timestamped snapshot under the backup root
  - writes `snapshot_manifest.json` and `checksums.sha256` for integrity evidence
- `scripts/ops/windows-single-host-restore-drill.ps1`
  - runs a non-destructive restore drill from a snapshot
  - verifies checksums and writes `restore_drill_report.json`

These scripts are the preferred operational path over manually typing repeated
NSSM and file-copy commands.

## Fresh install

Provision Fuseki first with `docs/ops/windows_fuseki_operator.md`, then continue with the API wheel install below.

### 1. Prepare directories

Run PowerShell as Administrator:

```powershell
$runtimeRoot = 'C:\Program Files\EarCrawler\runtime'
$workspaceRoot = 'C:\ProgramData\EarCrawler\workspace'
$configRoot = 'C:\ProgramData\EarCrawler\config'
$logRoot = 'C:\ProgramData\EarCrawler\logs'
$backupRoot = 'C:\ProgramData\EarCrawler\backups'

New-Item -ItemType Directory -Force -Path $runtimeRoot, $workspaceRoot, $configRoot, $logRoot, $backupRoot | Out-Null
New-Item -ItemType Directory -Force -Path (Join-Path $workspaceRoot 'kg\reports') | Out-Null
New-Item -ItemType Directory -Force -Path (Join-Path $workspaceRoot 'run\logs') | Out-Null
```

### 2. Install from the hermetic release bundle (authoritative release-grade path)

```powershell
$runtimeRoot = 'C:\Program Files\EarCrawler\runtime'
$releaseRoot = 'C:\ProgramData\EarCrawler\release\v0.2.5'
$venvPython = Join-Path $runtimeRoot '.venv\Scripts\python.exe'
$wheelPath = 'C:\path\to\earcrawler-0.2.5-py3-none-any.whl'
$checksumsPath = 'C:\path\to\checksums.sha256'
$hermeticZip = 'C:\path\to\hermetic-artifacts.zip'

py -3.11 -m venv (Join-Path $runtimeRoot '.venv')
& $venvPython -m pip install --upgrade pip

Expand-Archive -Path $hermeticZip -DestinationPath $releaseRoot -Force

pwsh (Join-Path $releaseRoot 'hermetic-artifacts\scripts\install-from-wheelhouse.ps1') `
  -PythonExecutable $venvPython `
  -LockFile (Join-Path $releaseRoot 'hermetic-artifacts\requirements-win-lock.txt') `
  -WheelhousePath (Join-Path $releaseRoot 'hermetic-artifacts\.wheelhouse') `
  -WheelPath $wheelPath `
  -ChecksumsPath $checksumsPath

& $venvPython -m pip show earCrawler
```

Treat this path as authoritative for production release installs. It is the same
dependency shape proven by `scripts/installed-runtime-smoke.ps1` in release
validation evidence.

#### Quick install fallback (not release-grade)

Use this only for local validation or emergency triage when the hermetic bundle
is not available:

```powershell
$runtimeRoot = 'C:\Program Files\EarCrawler\runtime'
$venvPython = Join-Path $runtimeRoot '.venv\Scripts\python.exe'

py -3.11 -m venv (Join-Path $runtimeRoot '.venv')
& $venvPython -m pip install --upgrade pip
& $venvPython -m pip install 'C:\path\to\earcrawler-0.2.5-py3-none-any.whl'
```

### 3. Set machine-scoped runtime configuration

The supported operator path uses machine environment variables so the service can be rebuilt without depending on per-user profiles or a source checkout.

```powershell
[System.Environment]::SetEnvironmentVariable('EAR_SYSTEM_INSTALL', '1', 'Machine')
[System.Environment]::SetEnvironmentVariable('EAR_TELEMETRY_CONFIG', 'C:\ProgramData\EarCrawler\config\telemetry.json', 'Machine')
[System.Environment]::SetEnvironmentVariable('EARCRAWLER_API_HOST', '127.0.0.1', 'Machine')
[System.Environment]::SetEnvironmentVariable('EARCRAWLER_API_PORT', '9001', 'Machine')
[System.Environment]::SetEnvironmentVariable('EARCRAWLER_FUSEKI_URL', 'http://127.0.0.1:3030/ear/query', 'Machine')
[System.Environment]::SetEnvironmentVariable('EARCRAWLER_REMOTE_LLM_POLICY', 'deny', 'Machine')
[System.Environment]::SetEnvironmentVariable('EARCRAWLER_ENABLE_REMOTE_LLM', '0', 'Machine')
[System.Environment]::SetEnvironmentVariable('EARCRAWLER_RETRIEVAL_MODE', 'dense', 'Machine')
[System.Environment]::SetEnvironmentVariable('EARCRAWLER_API_INSTANCE_COUNT', '1', 'Machine')
```

Only set these when you need them:

```powershell
[System.Environment]::SetEnvironmentVariable('EARCRAWLER_API_KEYS', 'ops=<replace-with-secret>', 'Machine')
[System.Environment]::SetEnvironmentVariable('TRADEGOV_API_KEY', '<replace-with-secret>', 'Machine')
[System.Environment]::SetEnvironmentVariable('GROQ_API_KEY', '<replace-with-secret>', 'Machine')
[System.Environment]::SetEnvironmentVariable('NVIDIA_NIM_API_KEY', '<replace-with-secret>', 'Machine')
```

If you enable optional remote answering, also set:

```powershell
[System.Environment]::SetEnvironmentVariable('EARCRAWLER_REMOTE_LLM_POLICY', 'allow', 'Machine')
[System.Environment]::SetEnvironmentVariable('EARCRAWLER_ENABLE_REMOTE_LLM', '1', 'Machine')
```

If you enable optional hybrid ranking, also set:

```powershell
[System.Environment]::SetEnvironmentVariable('EARCRAWLER_RETRIEVAL_MODE', 'hybrid', 'Machine')
```

If you enable optional local-adapter serving during an explicit review window,
also set:

```powershell
[System.Environment]::SetEnvironmentVariable('LLM_PROVIDER', 'local_adapter', 'Machine')
[System.Environment]::SetEnvironmentVariable('EARCRAWLER_ENABLE_LOCAL_LLM', '1', 'Machine')
[System.Environment]::SetEnvironmentVariable('EARCRAWLER_LOCAL_LLM_BASE_MODEL', 'Qwen/Qwen2.5-7B-Instruct', 'Machine')
[System.Environment]::SetEnvironmentVariable('EARCRAWLER_LOCAL_LLM_ADAPTER_DIR', 'C:\ProgramData\EarCrawler\models\<run_id>\adapter', 'Machine')
[System.Environment]::SetEnvironmentVariable('EARCRAWLER_LOCAL_LLM_MODEL_ID', '<run_id>', 'Machine')
```

Do not enable `EARCRAWLER_API_ENABLE_SEARCH` or `EARCRAWLER_ENABLE_KG_EXPANSION`
on deployed hosts. Those capabilities remain quarantined and are governed by
`docs/kg_quarantine_exit_gate.md` plus
`docs/capability_graduation_boundaries.md`.

Do not enable local-adapter serving on normal production-beta hosts. That track
is formally deprioritized for the current production-beta target by
`docs/local_adapter_deprioritization_2026-03-25.md`. Use it only for an
explicitly scoped review window with a named artifact and retained evidence.

Do not set `EARCRAWLER_ALLOW_UNSUPPORTED_MULTI_INSTANCE=1` on deployed hosts.
That override exists only for local experiments. The supported contract remains
one API service instance per host, and `/health` now reports that contract
explicitly.

### 4. Install the Windows service with NSSM

This guide assumes NSSM is installed at `C:\tools\nssm\nssm.exe`. If it is elsewhere, substitute that path.

```powershell
$nssm = 'C:\tools\nssm\nssm.exe'
$runtimeRoot = 'C:\Program Files\EarCrawler\runtime'
$workspaceRoot = 'C:\ProgramData\EarCrawler\workspace'
$logRoot = 'C:\ProgramData\EarCrawler\logs'
$venvPython = Join-Path $runtimeRoot '.venv\Scripts\python.exe'

& $nssm install EarCrawler-API $venvPython `
  '-m' 'uvicorn' 'service.api_server.server:app' '--host' '127.0.0.1' '--port' '9001'
& $nssm set EarCrawler-API AppDirectory $workspaceRoot
& $nssm set EarCrawler-API AppStdout (Join-Path $logRoot 'api-service.log')
& $nssm set EarCrawler-API AppStderr (Join-Path $logRoot 'api-service.log')
& $nssm set EarCrawler-API Start SERVICE_AUTO_START
```

Equivalent automation command:

```powershell
pwsh scripts/ops/windows-single-host-service.ps1 `
  -Action install `
  -NssmPath C:\tools\nssm\nssm.exe `
  -RuntimeRoot 'C:\Program Files\EarCrawler\runtime' `
  -WorkspaceRoot 'C:\ProgramData\EarCrawler\workspace' `
  -LogRoot 'C:\ProgramData\EarCrawler\logs' `
  -ApiHost 127.0.0.1 `
  -ApiPort 9001
```

Configure the service account and recovery policy in NSSM or Services:

- run as the intended least-privilege service account
- set recovery to restart on failure
- keep the service bound to the single host; do not claim multi-instance support

### 5. Start and verify

```powershell
C:\tools\nssm\nssm.exe start EarCrawler-API
Invoke-WebRequest http://127.0.0.1:9001/health -UseBasicParsing
Invoke-WebRequest `
  -Uri http://127.0.0.1:3030/ear/query `
  -Method Post `
  -ContentType 'application/sparql-query' `
  -Body 'SELECT (1 AS ?ok) WHERE { } LIMIT 1' `
  -UseBasicParsing
```

Equivalent automation commands:

```powershell
pwsh scripts/ops/windows-single-host-service.ps1 -Action start
pwsh scripts/ops/windows-single-host-service.ps1 -Action health -ApiHost 127.0.0.1 -ApiPort 9001
pwsh scripts/ops/windows-single-host-service.ps1 -Action status
```

Healthy install criteria:

- the service is running in `Get-Service EarCrawler-API`
- `/health` returns HTTP 200
- `/health` reports `runtime_contract.topology = single_host` and `runtime_contract.declared_instance_count = 1`
- `/health` reports `runtime_contract.capability_registry_schema = capability-registry.v1`
- `/health` reports `runtime_contract.capabilities.api.default_surface.status = supported`
- `/health` reports `runtime_contract.capabilities.api.search.status = quarantined`
- `/health` reports `runtime_contract.capabilities.kg.expansion.status = quarantined`
- the configured Fuseki query endpoint returns HTTP 200 to the trivial `SELECT`

Recommended post-install smoke for the supported baseline:

```powershell
pwsh scripts/api-smoke.ps1 -Host 127.0.0.1 -Port 9001 -ReportPath C:\ProgramData\EarCrawler\workspace\api_smoke.json
```

Treat failures in the supported API smoke report or the `/health` runtime
contract fields above as deployment blockers.

## Upgrade

Use the same dedicated virtual environment. Do not upgrade by copying a source checkout onto the host.

1. Copy the new signed wheel, checksum file, and `hermetic-artifacts.zip` to the host.
2. Stop the service.
3. Take a backup snapshot using the procedure below.
4. Reinstall from the hermetic wheelhouse payload into the existing virtual environment.
5. Restart the service.
6. Re-run the health checks.

Commands:

```powershell
$venvPython = 'C:\Program Files\EarCrawler\runtime\.venv\Scripts\python.exe'
$releaseRoot = 'C:\ProgramData\EarCrawler\release\v0.2.6'
$wheelPath = 'C:\path\to\earcrawler-0.2.6-py3-none-any.whl'
$checksumsPath = 'C:\path\to\checksums.sha256'
$hermeticZip = 'C:\path\to\hermetic-artifacts.zip'
C:\tools\nssm\nssm.exe stop EarCrawler-API
Expand-Archive -Path $hermeticZip -DestinationPath $releaseRoot -Force
pwsh (Join-Path $releaseRoot 'hermetic-artifacts\scripts\install-from-wheelhouse.ps1') `
  -PythonExecutable $venvPython `
  -LockFile (Join-Path $releaseRoot 'hermetic-artifacts\requirements-win-lock.txt') `
  -WheelhousePath (Join-Path $releaseRoot 'hermetic-artifacts\.wheelhouse') `
  -WheelPath $wheelPath `
  -ChecksumsPath $checksumsPath
C:\tools\nssm\nssm.exe start EarCrawler-API
Invoke-WebRequest http://127.0.0.1:9001/health -UseBasicParsing
```

If the release changes environment variables, update the machine-scoped values before restarting the service.

## Backup

Take backups before every upgrade and before any configuration or secret change.

### What to back up

- `C:\ProgramData\EarCrawler\config`
- `C:\ProgramData\EarCrawler\logs`
- `C:\ProgramData\EarCrawler\workspace`
- `%PROGRAMDATA%\EarCrawler\audit`
- `%PROGRAMDATA%\EarCrawler\spool`
- the currently installed wheel version and release checksum
- the service configuration and current machine environment values

If the Fuseki endpoint is local to the same host, back up its TDB2 data directory and service definition in the same maintenance window. EarCrawler does not automate that step.

### Backup procedure

Preferred automated path:

```powershell
pwsh scripts/ops/windows-single-host-backup.ps1 `
  -ProgramDataRoot 'C:\ProgramData\EarCrawler' `
  -RuntimeRoot 'C:\Program Files\EarCrawler\runtime' `
  -BackupRoot 'C:\ProgramData\EarCrawler\backups' `
  -RestartServiceAfterBackup
```

The command writes a snapshot directory with:

- `snapshot_manifest.json` (`windows-single-host-backup.v1`)
- `checksums.sha256`
- copied runtime state (`config`, `logs`, `workspace`, `audit`, `spool`)
- captured environment/service metadata where available

Manual equivalent:

```powershell
$stamp = Get-Date -Format 'yyyyMMdd-HHmmss'
$backup = "C:\ProgramData\EarCrawler\backups\$stamp"
$venvPython = 'C:\Program Files\EarCrawler\runtime\.venv\Scripts\python.exe'

C:\tools\nssm\nssm.exe stop EarCrawler-API
New-Item -ItemType Directory -Force -Path $backup | Out-Null

Copy-Item 'C:\ProgramData\EarCrawler\config' (Join-Path $backup 'config') -Recurse -Force -ErrorAction SilentlyContinue
Copy-Item 'C:\ProgramData\EarCrawler\logs' (Join-Path $backup 'logs') -Recurse -Force -ErrorAction SilentlyContinue
Copy-Item 'C:\ProgramData\EarCrawler\workspace' (Join-Path $backup 'workspace') -Recurse -Force -ErrorAction SilentlyContinue
Copy-Item "$env:PROGRAMDATA\EarCrawler\audit" (Join-Path $backup 'audit') -Recurse -Force -ErrorAction SilentlyContinue
Copy-Item "$env:PROGRAMDATA\EarCrawler\spool" (Join-Path $backup 'spool') -Recurse -Force -ErrorAction SilentlyContinue

& $venvPython -m pip show earCrawler | Set-Content (Join-Path $backup 'pip-show.txt')
reg export 'HKLM\SYSTEM\CurrentControlSet\Control\Session Manager\Environment' (Join-Path $backup 'machine-env.reg') /y | Out-Null
sc.exe qc EarCrawler-API > (Join-Path $backup 'service-qc.txt')
sc.exe qfailure EarCrawler-API > (Join-Path $backup 'service-failure.txt')
```

After the backup completes, either keep the service stopped for maintenance or start it again:

```powershell
C:\tools\nssm\nssm.exe start EarCrawler-API
```

## Restore

Restore is for rebuilding the same version and state after host loss or operator error.

1. Recreate the standard host layout.
2. Recreate the Python virtual environment.
3. Reinstall the same wheel version captured in the backup.
4. Restore `config`, `logs`, `workspace`, `audit`, and `spool`.
5. Reapply the required machine environment variables from the reviewed backup record.
6. Recreate the NSSM service.
7. Start the service and run the health checks.

If the original host used a local Fuseki instance, restore that dataset before you start EarCrawler.

Before a maintenance window, run a non-destructive restore drill from the latest
snapshot and archive the report:

```powershell
pwsh scripts/ops/windows-single-host-restore-drill.ps1 `
  -SnapshotPath 'C:\ProgramData\EarCrawler\backups\<snapshot-id>' `
  -DrillRoot 'C:\ProgramData\EarCrawler\backups\drills\<snapshot-id>'
```

Treat a failing drill report (`status: fail`) as a release blocker for host
upgrade/rollback operations.

### Recurring DR evidence automation

To make backup validity and restore integrity evidence periodic instead of
ad hoc, schedule the recurring DR evidence runner:

```powershell
pwsh scripts/ops/windows-recurring-dr-evidence.ps1 `
  -ProgramDataRoot 'C:\ProgramData\EarCrawler' `
  -RuntimeRoot 'C:\Program Files\EarCrawler\runtime' `
  -ApiBackupRoot 'C:\ProgramData\EarCrawler\backups' `
  -FusekiProgramDataRoot 'C:\ProgramData\EarCrawler\fuseki' `
  -FusekiHome 'C:\Program Files\Apache\Jena-Fuseki-5.3.0' `
  -FusekiBackupRoot 'C:\ProgramData\EarCrawler\backups\fuseki' `
  -EvidenceRoot 'C:\ProgramData\EarCrawler\backups\recurring-evidence' `
  -RetentionRuns 30 `
  -SkipServiceControl
```

This runner performs:

- API backup snapshot (`windows-single-host-backup.ps1`)
- API non-destructive restore drill (`windows-single-host-restore-drill.ps1`)
- Fuseki backup snapshot (`windows-fuseki-backup.ps1`)
- Fuseki non-destructive restore drill (`windows-fuseki-restore-drill.ps1`)
- run summary at
  `C:\ProgramData\EarCrawler\backups\recurring-evidence\dr-evidence-run-<run_id>.json`
- retained run index at
  `C:\ProgramData\EarCrawler\backups\recurring-evidence\dr-evidence-index.json`

## Observability and retained evidence

Keep the following host-local evidence with each install, upgrade, or rollback:

- wheel filename, checksum file, and release version
- `/health` output captured after service start
- supported API smoke report for the installed host
- backup `snapshot_manifest.json` and `checksums.sha256`
- restore drill report when a drill was run
- recurring DR evidence run report and retained run index when scheduled
- service log output under `C:\ProgramData\EarCrawler\logs`

Telemetry is optional, but the host should still preserve enough local evidence
to answer: what version was installed, what runtime contract was active, what
capability posture was active, and whether the supported smoke checks passed.

If telemetry is enabled, treat `C:\ProgramData\EarCrawler\config\telemetry.json`
and the spool/audit directories as part of the deployment state and back them
up with the rest of the host evidence.

## Optional capability playbook

Use this section only for explicit operator-owned validation windows. It does
not change capability status by itself.

Release-shaped smoke command:

```powershell
pwsh scripts/optional-runtime-smoke.ps1 `
  -Host 127.0.0.1 `
  -Port 9001 `
  -SkipLocalAdapter `
  -ReportPath C:\ProgramData\EarCrawler\workspace\kg\reports\optional-runtime-smoke.json
```

What this proves:

- search gate default-off behavior (`/v1/search` returns 404)
- search local-validation opt-in behavior (`EARCRAWLER_API_ENABLE_SEARCH=1`)
- search rollback behavior (disabled again)
- KG expansion failure-policy behavior (`disable` and `error`) with deterministic checks

For the operator-owned production-like proof package required before any search
or KG promotion decision, also run:

```powershell
pwsh scripts/search-kg-prodlike-smoke.ps1 `
  -Host 127.0.0.1 `
  -Port 9001 `
  -FusekiHost 127.0.0.1 `
  -FusekiPort 3040 `
  -ReportPath C:\ProgramData\EarCrawler\workspace\kg\reports\search-kg-prodlike-smoke.json
```

This validation path is still quarantined. It uses the optional
text-index-enabled Fuseki workflow from `docs/ops/windows_fuseki_operator.md`
and proves only that the real operator-owned API + Fuseki runtime shape can:

- seed a deterministic validation graph into a text-index-backed Fuseki dataset
- return `200` from `/v1/search` when `EARCRAWLER_API_ENABLE_SEARCH=1`
- execute Fuseki-backed KG expansion successfully when `EARCRAWLER_ENABLE_KG_EXPANSION=1`

Treat a failure in this proof as a no-go signal for promotion evidence, not as
a supported-baseline outage.

If you have a real Task 5.3 adapter artifact and are explicitly resuming the
deprioritized local-adapter track for a review window, run:

```powershell
pwsh scripts/optional-runtime-smoke.ps1 `
  -Host 127.0.0.1 `
  -Port 9001 `
  -LocalAdapterRunDir C:\ProgramData\EarCrawler\models\<run_id> `
  -ReportPath C:\ProgramData\EarCrawler\workspace\kg\reports\optional-runtime-smoke.json
```

For a release-candidate adapter review window, also retain the evidence bundle
defined in `docs/local_adapter_release_evidence.md`:

## Rollback

Rollback is the fast path for a bad release when the prior host state is still available.

1. Stop the service.
2. Reinstall the last known-good wheel from the matching hermetic release payload.
3. Reapply the prior machine environment values if the failed release changed them.
4. Restore the prior `config` or `workspace` snapshot if the failed release mutated them.
5. Start the service.
6. Verify `/health`.

Commands:

```powershell
$venvPython = 'C:\Program Files\EarCrawler\runtime\.venv\Scripts\python.exe'
$releaseRoot = 'C:\ProgramData\EarCrawler\release\v0.2.5'
$wheelPath = 'C:\path\to\earcrawler-0.2.5-py3-none-any.whl'
$checksumsPath = 'C:\path\to\checksums.sha256'
$hermeticZip = 'C:\path\to\hermetic-artifacts.zip'
C:\tools\nssm\nssm.exe stop EarCrawler-API
Expand-Archive -Path $hermeticZip -DestinationPath $releaseRoot -Force
pwsh (Join-Path $releaseRoot 'hermetic-artifacts\scripts\install-from-wheelhouse.ps1') `
  -PythonExecutable $venvPython `
  -LockFile (Join-Path $releaseRoot 'hermetic-artifacts\requirements-win-lock.txt') `
  -WheelhousePath (Join-Path $releaseRoot 'hermetic-artifacts\.wheelhouse') `
  -WheelPath $wheelPath `
  -ChecksumsPath $checksumsPath
C:\tools\nssm\nssm.exe start EarCrawler-API
Invoke-WebRequest http://127.0.0.1:9001/health -UseBasicParsing
```

Rollback is not complete until:

- the prior wheel version is installed
- the service is running
- `/health` is green
- any local Fuseki dependency is also back on the prior known-good state

## Secret rotation

The authoritative deployment path uses machine-scoped environment variables for service secrets. Rotate secrets there, then restart the affected service or job host.

### Rotate inbound API keys

```powershell
[System.Environment]::SetEnvironmentVariable('EARCRAWLER_API_KEYS', 'ops=<new-secret>', 'Machine')
C:\tools\nssm\nssm.exe restart EarCrawler-API
```

Clients must then send `X-Api-Key: ops:<new-secret>` or the unlabeled raw secret form if you choose not to use labels.

### Rotate upstream or provider secrets

Examples:

```powershell
[System.Environment]::SetEnvironmentVariable('TRADEGOV_API_KEY', '<new-secret>', 'Machine')
[System.Environment]::SetEnvironmentVariable('GROQ_API_KEY', '<new-secret>', 'Machine')
[System.Environment]::SetEnvironmentVariable('NVIDIA_NIM_API_KEY', '<new-secret>', 'Machine')
C:\tools\nssm\nssm.exe restart EarCrawler-API
```

If a secret is no longer needed, remove it explicitly:

```powershell
[System.Environment]::SetEnvironmentVariable('GROQ_API_KEY', $null, 'Machine')
```

### Rotate optional remote-answering posture

To disable remote answering again:

```powershell
[System.Environment]::SetEnvironmentVariable('EARCRAWLER_REMOTE_LLM_POLICY', 'deny', 'Machine')
[System.Environment]::SetEnvironmentVariable('EARCRAWLER_ENABLE_REMOTE_LLM', '0', 'Machine')
C:\tools\nssm\nssm.exe restart EarCrawler-API
```

## Still-missing deployment evidence

This guide is the authoritative operator path for the supported Windows
single-host baseline, but it still does not by itself prove:

- multi-instance or load-balanced correctness
- direct broader-than-loopback exposure without the separate front-door pattern
- shared-state rollout, drain, or cache-coordination behavior across instances

Treat those as out of scope until separate design, validation, and retained
evidence exist.

## Operational notes

- Keep at least the current and prior signed wheels, checksum files, and `hermetic-artifacts.zip` bundles on the host or in the release vault.
- Record the exact wheel filename and backup snapshot id used for every upgrade and rollback.
- Treat `C:\ProgramData\EarCrawler\config\telemetry.json` as part of the deployment state if telemetry is enabled.
- For feature-only rollback, reset `EARCRAWLER_RETRIEVAL_MODE` to `dense` and
  clear any `local_adapter` env vars before rolling back the wheel.
- Do not use this guide to claim container or multi-instance support. The supported target remains one Windows host.
- If scale-out becomes in scope later, start from `docs/ops/multi_instance_deferred.md` instead of inferring support from this guide.
- Broader-than-loopback exposure uses a separate front door. The reference pattern is IIS + ARR + Windows Authentication with EarCrawler still bound to loopback; see `docs/ops/external_auth_front_door.md`, `scripts/ops/iis-earcrawler-front-door.web.config.example`, and `scripts/ops/iis-front-door-smoke.ps1`.
- When the optional IIS front door is deployed, retain the resulting `iis-front-door-smoke.json` report with the same host-local evidence set as install, upgrade, and rollback records.
